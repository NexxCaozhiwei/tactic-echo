# P5.4 实机验收：P4.3 当前动作栏路由、钢条稳定与打断冷却

## 部署

1. 覆盖项目根目录与 WoW 实际加载目录中的 `Interface/AddOns/!TacticEcho`。
2. 重启 TEK，游戏内执行：

```text
/reload
/tecache refresh
```

本版未修改 TEK，无需重建 `TEK.exe`。

## 验收 A：普通可打断读条

使用当前实际绑定的打断按钮测试。预期：约 3 个观察样本且首样本后约 0.40 秒，出现：

```text
reason = compat_interrupt_candidate
routeMode = action_info_represented_spell_compat   # 仅当宏正文不可读
macroOpaqueRepresentedSpell = true                 # 同上
bindingToken > 0
dispatchOrigin = reaction
```

宏正文可正常恢复时，可能显示既有 direct/macro routeMode；不要求特定宏名。

## 验收 B：真实钢条

不得出现 reaction candidate。优先检查：

```text
unit_event_not_interruptible
native_visible_shield
compat_native_steel_suspected
compatibility_evidence_settling
compatibility_steel_settling
```

`showShield`/`barType` 本身不是结论。若某类钢条始终只有完全未知数据，请上传新的 `!TacticEcho.lua`，不要将未知状态当作已验证钢条证据。

## 验收 C：打断技能自身 CD

让实际打断按钮进入非 GCD 自身冷却。预期：

```text
reason = interrupt_action_cooldown
bindingToken = 0
cooldownActive = true
cooldownOnGCD = false
cooldownActionSlot = <当前实际动作条槽位>
```

此时主键应回到官方正常推荐，不能持续显示打断技能或产生 reaction TEAP 帧。

## 回传

上传新的 `!TacticEcho.lua`。若 AddOn 已出现 `dispatchOrigin=reaction` 但游戏内未施放，再补充 TEK 诊断包。
