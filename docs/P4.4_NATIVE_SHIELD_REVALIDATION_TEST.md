# P4.4 严格钢条判定实机验收

## 前置条件

- 当前 AddOn 为 `1.0.41`，游戏内 `/reload` 后确认加载。
- 使用已验证 P4.3 反应派发能力的 TEK.exe；P4.4 不改 TEK 输入代码。
- 自动打断已开启，TE 为“运行中”，打断技能已有 P2 安全路由和真实键位。
- 先关闭自动爆发，避免 `Burst占用` 影响观察。

## 观察位置

设置中心 → 打断与控制 → 打断设置 → “当前打断 / 控制监控”。

重点字段：

- `判定=`：资格来源或阻断原因。
- `原生=`：P4 读取到的原生证据。
- `视觉采样=1/2`、`2/2`：原生无盾/明确原生可打断路径的连续样本。
- `盾组件=`：实际被读取的 shield 子组件；仅这个可见状态具有视觉判定资格。

## 场景 A：普通可打断读条

预期路径之一：

```text
判定=unit_api_confirmed
```

或：

```text
判定=unit_event_interruptible_confirmed
```

或原生视觉兜底：

```text
判定=native_visual_settling / 原生=native_visible_no_shield:... / 视觉采样=1/2
→
判定=visible_no_shield_confirmed / 视觉采样=2/2
→
confirmed_interrupt_candidate
```

实际按键只应发送一次；读条尚未结束时 reaction candidate 可持续可见，但 TEK 不能重复输入。

## 场景 B：明确钢条 / 不可打断读条

预期：不产生 reaction TEAP 及按键。诊断至少应出现其一：

```text
判定=unit_api_not_interruptible
判定=unit_event_not_interruptible
判定=native_visible_shield
```

`showShield=true` 单独出现不能作为钢条结论。

## 场景 C：读条存在但证据未知

预期：只高亮，不派发。

```text
判定=interruptibility_unconfirmed
```

这不是 TEK 拒绝，也不是键位问题；代表 P4.4 没有获得足够的可打断证据。

## 场景 D：安全路由不存在

预期：

```text
判定=... / 路由=interrupt_binding_route_missing
```

这表示读条资格已通过，但打断技能没有当前可见默认动作条上的安全 BindingToken 路由。P4 最多重扫一次现有缓存，不修改绑定、不创建按键。

## 建议上传材料

若判断异常，提供同一次测试的：

1. `WTF/Account/.../SavedVariables/!TacticEcho.lua`；
2. TEK diagnostics zip；
3. 说明这是普通读条还是明确钢条，是否真的发生过打断。
