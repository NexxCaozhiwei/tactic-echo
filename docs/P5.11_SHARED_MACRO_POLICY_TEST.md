# P5.11 共享宏资格与控制/防御/生存 HUD 实机验收

## 准备

在当前可见 Blizzard 默认动作条中保留以下玩家已有来源，不要让插件创建或修改宏：

```text
#showtooltip
/cast [@cursor] 冰冻陷阱
```

将该宏放在 `CTRL+1`。可在其他位置保留冰冻陷阱直放或同技能宏，用于确认不跨槽位替代。另准备一个既有生存物品宏，例如 `/use item:5512`；可保留已有条件、目标分支或 `/castsequence` 结构。

## 验收步骤

1. `/reload` 后执行 `/tecache refresh`。控制卡的冰冻陷阱来源必须只指向 `CTRL+1` 的当前按钮与槽位；若当前栏位身份无法确认，HUD 保持显示但阻断，不能改点到别的冰冻陷阱槽位。
2. 控制 HUD 左键点击应等价于鼠标点击该原生 `CTRL+1` 动作条宏；`[@cursor]` 仍由原宏决定。动作码与 BindingToken 保持 0，不能产生 TEAP/TEK 控制派发。
3. 查看控制诊断：无关 `BUTTON3` / “坐骑”宏不能列在胁迫、冰冻陷阱的“未匹配宏候选”。若同一 `CTRL+1` 有效 macro index 的正文临时不可读，而动作条文本仍命名冰冻陷阱，可显示该**同一 CTRL+1**失败记录，但不允许执行。
4. 验证防御宏：已有条件分支、`@player`、目标管理或 `/castsequence` 的当前可见宏，身份与正文确认后可作为手动 HUD 来源；无可用快捷键时仍固定 `BindingToken=0`。
5. 验证生存 `/use` 宏：`/use item:5512`、`/use 5512`、`/use 治疗石` 或现有条件分支只在当前宏身份和正文都确认时提供 HUD 手动入口。宏名、图标和背包/宏列表存在不足以建立来源。
6. 保留 P4 回归：正文不可读、但 action-info 明确表示打断技能的当前宏可在已有真实 BindingToken 下保留 target-only reaction transport；它不能变成控制/防御/生存 HUD 来源，不能推断 focus/mouseover/cursor，也不能进入 AutoBurst。
7. 回归 P5.8/P5.10：主键启停、HUD CD 数字、DurationObject 转盘、标签、排序、HUD/原生动作条 `manual_hold` 优先正常；自动打断仍为 `auto_interrupt_suspended`；任何脱战帧都没有 Burst plan/capture/TEAP/TEK。
