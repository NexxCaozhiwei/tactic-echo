# P5.5 双 action-info 宏身份实机验收

## 前提

- 覆盖 AddOn，并确认游戏内 `/reload`；
- 执行 `/tecache refresh`；
- 自动打断与“兼容未确认读条派发”已启用；
- 使用当前动作栏实际绑定的打断按钮，不需要修改或重命名任何宏；
- 不需要重建 `TEK.exe`。

## 1. 正常可打断读条

预期在同一读条完成约 3 次共享观察和 0.40 秒稳定后，出现：

```text
reason = compat_interrupt_candidate
dispatchOrigin = reaction
routeMode = action_info_macro_spell_compat
```

若客户端采用另一 action-info 形态，`routeMode` 可为：

```text
action_info_represented_spell_compat
```

两者都只表示当前动作栏按钮的 target-only transport。

## 2. 真实钢条

不得出现 reaction candidate 或非零 reaction Token。日志应优先表现为：

```text
unit_api_not_interruptible
native_visible_shield
unit_event_not_interruptible
native_not_interruptible
compat_native_steel_suspected
```

若客户端没有任何可读钢条证据，日志可能仍为兼容稳定状态；上传 SavedVariables 继续核对。

## 3. 打断技能自身 CD

将当前实际打断按钮置于非 GCD 自身 CD 后，敌方读条时应记录：

```text
reason = interrupt_action_cooldown
bindingToken = 0
cooldownActionSlot = <实际槽位>
```

主键必须回到官方正常序列，不应持续显示打断技能，也不得有 reaction TEAP 帧。

## 上传字段

上传新的 `!TacticEcho.lua`。重点核对：

```text
routeMode
macroOpaqueRepresentedSpell
compatibilityEvidenceSamples
compatibilityEvidenceAge
cooldownKnown
cooldownActive
cooldownActionSlot
dispatchOrigin
```
