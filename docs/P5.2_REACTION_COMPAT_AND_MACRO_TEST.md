# P5.2 实机验收：兼容未确认读条派发与宏命中守卫

## 安装

1. 覆盖项目根目录补丁文件。
2. 覆盖 WoW 实际加载目录 `Interface/AddOns/!TacticEcho`。
3. 重启 TEK；本版未修改 TEK，不需要重新构建 `TEK.exe`。
4. 进入游戏后执行：

```text
/reload
/tecache refresh
/temapping p52_reaction_compat
```

在“打断设置”中确认：

- `启用自动打断` 已打开；
- `兼容未确认读条派发` 已打开；
- 反制射击或当前职业打断技能已显示真实键位。

## 场景 A：P4.3 兼容候选复测

目标或鼠标指向出现真实读条、但面板仍显示 `interruptibility_unconfirmed` 时：

```text
autoReactionRuntime.last.reason = compat_interrupt_candidate
confirmationReason = compat_active_cast_no_hard_steel
compatibilityFallback = true
bindingToken > 0
```

TEK trace 应出现 `dispatch_origin=reaction`，且只会对同一稳定 sequence 成功派发一次。

若关闭“兼容未确认读条派发”，同一场景应恢复：

```text
reason = interruptibility_unconfirmed
bindingToken = 0
```

## 场景 B：明确钢条仍不打

分别测试真实可见盾、明确不可打断 API 或 `NOT_INTERRUPTIBLE` 事件。预期：

```text
native_visible_shield
unit_api_not_interruptible
unit_event_not_interruptible
native_not_interruptible
```

任一情况下均不得出现 `dispatch_origin=reaction` 或非零 reaction BindingToken。

## 场景 C：中文宏名与反制射击

将反制射击多行宏命名为 `打断`、`DDD`、或任意唯一/同名中文名称；分别刷新缓存。

```lua
#showtooltip
/stopcasting
/cast [@focus,nodead] 反制射击
/cleartarget
/targetenemy
/cast 反制射击
/targetlasttarget
```

映射应仍能识别宏身份。若 Retail 返回代表 SpellID `147362`，导出中应能看到
`representedSpellID=147362` 与唯一语义身份来源；不得保存宏正文。

## 场景 D：焦点优先选敌回退宏的自动范围

上面的多行宏：

- 焦点敌人正在读条：可以作为焦点路径进入 candidate；
- 当前目标正在读条、焦点不存在：不会自动使用后续 `/targetenemy` 分支，诊断应为
  `macro_managed_target_fallback_target_unverifiable`；
- 给反制射击额外放置普通当前目标按钮并绑定键位后：当前目标读条应优先使用该直接动作条绑定。

这是为了避免 `/targetenemy` 在按键后选择到与 P3 当前目标不同的敌人。
