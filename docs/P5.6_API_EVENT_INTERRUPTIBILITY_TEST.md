# P5.6 实机验收：API/事件确认自动打断

## 前置条件

- 覆盖 P5.6 AddOn 后执行 `/reload` 与 `/tecache refresh`；
- 自动打断已启用；
- 打断按钮或其已有宏在当前可见 Blizzard 默认动作条，且有真实键位；
- TE 处于 `armed`，角色在战斗中；
- 当前动作栏实际打断按钮不在自身 CD。

## 1. 普通可打断读条

预期日志：

```text
reason = confirmed_interrupt_candidate
confirmationReason = unit_api_confirmed
# 或
confirmationReason = unit_event_interruptible_confirmed

dispatchOrigin = reaction
bindingToken > 0
```

不再出现 `compat_interrupt_candidate`。若 Unit API 返回 `notInterruptible=false`，不需要等待兼容采样窗口。

## 2. 钢条

预期：不出现 reaction candidate；常见理由之一：

```text
unit_api_not_interruptible
unit_event_not_interruptible
native_visible_shield
```

## 3. API/事件都未知

预期：只观察、不占主键：

```text
reason = interruptibility_unknown
bindingToken = 0
```

这不是派发失败，而是严格模式拒绝把未知状态当作可打断。

## 4. 打断技能自身 CD

在目标进行已确认可打断读条时，让实际打断按钮进入非 GCD 自身 CD。

预期：

```text
reason = interrupt_action_cooldown
bindingToken = 0
cooldownActive = true
cooldownOnGCD = false
```

主键应回到官方正常序列，不得显示或派发 reaction。

## 上传材料

如仍异常，上传新的 `!TacticEcho.lua`；重点检查 `confirmationReason`、`eventEvidence`、`routeMode`、`cooldownActionSlot`、`cooldownKnown`、`cooldownActive` 与 `bindingToken`。
