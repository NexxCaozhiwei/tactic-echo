# Tactic Echo 1.0.18 基线：持久爆发序列与全 HUD 物品 GCD 隔离

## 目标

解决两类实机问题：

1. 13/14 槽饰品在公共 GCD 中仍会显示短倒数或转盘；
2. 前置/后置 AutoBurst 在饰品或技能被打断、官方推荐旋转、窗口冷却读取短暂冲突后，可能因旧的重校验时限而停在序列中。

本版本把可派发的 Burst 步骤与普通官方推荐统一为“新鲜候选持续发布、TEK 全局频率统一限流、精确证据才推进”的模型。

## 已核实的现场链路

上传的 SavedVariables 中，规则 `manual:pre:simple:343527:trinket:13:false` 已创建计划；13 槽物品 `193701` 在恢复期被精确确认，随后窗口 `343527` 因旧版 `spell_cooldown_active` 重校验超时而中止。该记录证明卡住点不是饰品绑定，也不是计划未进入，而是旧的“窗口官方推荐旋转 + 冷却冲突 + 重校验超时”终止路径。

## 固定合同

### 1. HUD 冷却展示

1. 所有主 HUD、Burst、打断、控制、防御、药水等卡片的 CD 绘制统一经过 `UI/TacticalIconButton.lua`；不得在其他 HUD 模块单独创建 Cooldown 绘制路径。
2. 任何物品来源卡片（`itemID`，或临时仍只保留的 13/14 稳定装备槽位/物品类别；包含饰品与药水）均不得读取原生 ActionBar/Item `DurationObject` 作为 HUD 主 CD。原生对象可能实际代表共享 `61304` GCD。
3. 物品卡只显示 `CooldownResolver` 已归一化且确认非 GCD 的自身 CD 数值；公共 GCD 的主 CD、数字、`gcdCooldown` 数值层和 `61304` 原生覆盖层全部隐藏。
4. 普通 SpellID 卡继续保留既有原生 DurationObject 与 GCD 展示策略。
5. 对变体/覆盖动作条，`IconState` 读取受信任动作条的公共 `isOnGCD` 布尔值；若基础 Spell API 短暂报告 active、动作条实际仅在 GCD，则强制归类为 GCD，而非自身 CD。

### 2. Burst 序列

1. 一旦以官方窗口 SpellID 建立计划，窗口步骤与注入步骤即绑定到该 `planId`；官方推荐后续旋转只记录 `official_window_departed_plan_latched`，不释放、不取消已创建计划。
2. 每个仍可派发的逻辑步骤在 `READY_NOW`、`QUEUE_WINDOW`、`GCD_LOCKED` 持续生成新鲜 `armed` TEAP sequence；物理输入次数只受 TEK 当前全局 dispatch rate、前台、Hook、手动让权和新鲜度门禁限制。
3. 无外层计划超时、无固定重试次数、无确认/重校验/恢复期的终止倒计时。各时间常量只用于诊断心跳和恢复标记。
4. 前置饰品未确认后进入持久恢复；自动候选继续发布，同时监听锁定 `inventorySlot + ItemID` 的非 GCD 自身 CD。手动补按出现该精确 CD 后立即推进窗口。
5. API 冷却读取 `UNKNOWN` 不再把已绑定步骤变成观察帧：只要 GCDGate 能提供公共阶段，仍按同一全局频率持续发布候选；UNKNOWN 永远不是成功、跳过或步骤推进证据。
6. 规则跳过仅接受相关步骤的可验证非 GCD 自身 CD。前置与后置简易注入，以及窗口自身 CD 处置，均需两次独立、身份一致的实时证据。共享 GCD、图标灰度、Buff、动作条阴影、未知读数和其他物品 CD均不能跳过。
7. 绑定失效、装备变化、脱战、死亡/载具等硬阻断仍 fail-closed，并对已创建计划保留窗口离开锁。

## 实机复测

1. 13/14 槽饰品 ready 时连续施放普通 GCD 技能：任何 HUD 卡片上的该物品都不得出现约 1～1.5 秒公共 GCD 数字或转盘。
2. 使用具有真实自身 CD 的饰品：仅显示其真实物品 CD；公共 GCD 不得叠加第二层转盘。
3. 前置与后置各测试一次：在 GCD、普通读条、短暂停顿或饰品首次未被游戏接收时，观察同一 Burst 步骤持续产生新 sequence；TEK 实际输入次数仅随全局频率设置变化。
4. 前置饰品被中断多次后，手动补按同一锁定饰品，确认精确 CD 出现后必须自动进入窗口候选。
5. 进入窗口后官方推荐立即改为其他技能：已创建计划仍应继续当前步骤；窗口技能只在确认成功、经规则确认自身 CD，或硬阻断时结束。
