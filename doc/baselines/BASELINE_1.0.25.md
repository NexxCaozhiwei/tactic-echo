# Tactic Echo 1.0.25 基线：注入预检组装

## 问题背景

射击猎人规则 `乱射 260243（窗口） → 百发百中 288613（注入）` 的第二轮实测表明：当 `288613` 已处于自身冷却时，旧逻辑仍可能先创建 Burst 计划，再持续重新派发 X 并等待不可靠的 CD 或失败回执。前两版扩展“计划创建后跳过”的证据来源，仍无法覆盖该客户端的实际冷却返回。

## 修复目标

将注入技能或饰品的可用性判断前移到**窗口命中后的计划创建之前**：

```text
检测官方窗口
→ 按当前规则顺序读取每个注入技能 / 饰品的自身 CD
→ 仅明确可用（或仅共享 GCD）的项可以加入本轮计划
→ 自身 CD、确认中、未知冷却、无绑定的项直接排除
→ 至少一项可用：按简易 / 集中规则创建序列
→ 全部排除：不创建 Burst 计划，官方窗口正常继续
```

Profile fallback 的多个注入 SpellID 会按配置顺序筛选第一个可用项；手动技能或手动饰品规则仍只有当前配置的一项。

## 瞬时变化保护

若注入在建计划后的四帧 handoff 期间，或首次候选尚未确认成功时，重新变为明确 CD 或冷却未知：

```text
释放未确认 Burst plan
→ 不锁窗口离开
→ 不依据失败事件推断 CD
→ 不再重复派发注入
→ 交还官方窗口
```

## 不变项

- 普通官方技能、Session Policy、UI 运行标签、TEAP / TEK 协议、BindingToken 与输入频率不改变。
- 共享 GCD 不是自身 CD；明确“仅共享 GCD”仍可参与计划。
- 不以 `UNIT_SPELLCAST_FAILED`、灰色遮罩、HUD DurationObject、资源、目标、距离或宏猜测作为 CD 结论。
- 若没有任何可用注入，系统不把官方窗口 C 伪装成 Burst 计划中的窗口步骤。

## 回归要求

1. X 已自身 CD 时，C 窗口不创建计划、不输出 X 候选；
2. Profile 候选 A 冷却、B 可用时，选择 B；全部候选冷却时不建计划；
3. 共享 GCD 仍允许已就绪注入排队；
4. 冷却未知 / 确认中不得进入计划；
5. 注入在 handoff 或首次确认前变为未知 / CD 时，释放计划而不反复重发；
6. 起手 bridge、四帧 handoff、战斗内原有序列、TEK 全局门禁均不退化。
